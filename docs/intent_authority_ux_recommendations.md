# Intent Authority UX Recommendations

Prepared: April 17, 2026

Related implementation document: `docs/intent_authority_map_implementation_plan.md`

## Goal

Reframe the routing-intent experience around operator questions instead of model families.

The four questions the UI should answer without manual assembly are:

1. **What intent is authoritative** for each managed prefix or ASN right now?
2. **How does runtime state compare** to that authority, and where are the gaps?
3. **What changed** in authority since the last derivation, and why?
4. **Has the workflow completed** — derivation through apply, lint, simulation, and verification — or did a stage stall?

The plugin already computes and stores enough metadata to answer all four. But the UI remains organized around object families — profiles, bindings, derivation runs, intents, reconciliation runs, intent results, published results, match rows, change plans, lint runs, simulation runs, write executions, rollback bundles, and exceptions. That forces every answer to be assembled by the operator across multiple list and detail pages.

### What the pipeline produces per subject

For a single managed prefix, the pipeline can generate the following artifacts during a complete cycle:

1. **Authority definition**: `ROAIntent` with `derivation_run`, `intent_profile`, `source_rule`, `applied_override`, `derived_state`, `exposure_state`, `explanation`, `summary_json` (containing `profile_context_group_names`, `binding_context_groups`, `warnings`, `delegated_scope`).
2. **Reconciliation evidence**: `ROAIntentResult` with `result_type` (12 values: MATCH, MISSING, ASN_MISMATCH, ASN_AND_MAX_LENGTH_OVERBROAD, ASN_AND_MAX_LENGTH_TOO_NARROW, ASN_AND_MAX_LENGTH_MISMATCH, PREFIX_MISMATCH, MAX_LENGTH_OVERBROAD, MAX_LENGTH_TOO_NARROW, STALE, INACTIVE_INTENT, SUPPRESSED_BY_POLICY), `severity` (INFO/WARNING/ERROR/CRITICAL), `best_roa_object` or `best_imported_authorization`, `match_count`, `details_json`.
3. **Published-side evidence**: `PublishedROAResult` with `result_type` (11 values: MATCHED, ORPHANED, DUPLICATE, BROADER_THAN_NEEDED, MAX_LENGTH_TOO_NARROW, WRONG_ORIGIN, WRONG_ORIGIN_AND_MAX_LENGTH_OVERBROAD, WRONG_ORIGIN_AND_MAX_LENGTH_TOO_NARROW, WRONG_ORIGIN_AND_MAX_LENGTH_MISMATCH, STALE, UNSCOPED), `severity`, `matched_intent_count`, `details_json`.
4. **Match candidates**: `ROAIntentMatch` with `match_kind` (EXACT, ORIGIN_CONFLICT, PREFIX_CONFLICT, LENGTH_BROADER, LENGTH_NARROWER, SUPERSET, SUBSET, STALE_CANDIDATE), `is_best_match`, `details_json`.
5. **Lint quality gate**: `ROALintFinding` with `finding_code`, `severity`, `details_json`, linked to a `ROALintRun` per reconciliation or per change plan.
6. **Change plan action**: `ROAChangePlanItem` with `action_type`, `plan_semantic`, `provider_operation`, `before_state_json`, `after_state_json`, `reason`.
7. **ROV simulation**: `ROAValidationSimulationResult` with `outcome_type` (VALID, INVALID, NOT_FOUND), `approval_impact`, `scenario_type`, `details_json`.
8. **Governance**: `ApprovalRecord` with `disposition`, `recorded_by`, `ticket_reference`, `simulation_review_json`.
9. **Provider write**: `ProviderWriteExecution` with `execution_mode`, `status`, `request_payload_json`, `response_payload_json`.
10. **Rollback**: `ROAChangePlanRollbackBundle` with `rollback_delta_json`, `status`, governance metadata.

All of this exists per subject but is spread across 10+ detail pages with no aggregation layer.

ASPA intents are globally scoped to an organization and intent_key. They have no derivation run, no profile, no rule, no override, no derived state, and no structured summary. Only ROA is ready for the full authority-explainer pages in the first phase.

### Template binding lifecycle affecting authority freshness

`RoutingIntentTemplateBinding.state` cycles through four values:

- **CURRENT**: Binding compilation matches the current template fingerprint. Authority from this binding is current.
- **STALE**: Template has been modified since last compilation. Authority from this binding is outdated.
- **PENDING**: Binding awaits first compilation or explicit regeneration. No authority yet.
- **INVALID**: Compilation failed. Authority from this binding is unreliable.

Any authority-oriented page must surface binding staleness because a STALE or INVALID binding means the current derivation may not reflect the template author's latest intent.

### Models the original analysis did not address

The following subsystems affect the authority and drift UX but were not analyzed:

- **Lint subsystem**: `ROALintRun`, `ROALintFinding`, `ROALintAcknowledgement`, `ROALintSuppression`, `ROALintRuleConfig`. Lint runs gate change plans. A plan with unacknowledged blocking findings cannot proceed to apply. This is a critical workflow stage between reconciliation and execution.
- **Simulation subsystem**: `ROAValidationSimulationRun`, `ROAValidationSimulationResult`. Simulates BGP ROV outcomes for proposed changes. An `overall_approval_posture` of BLOCKING prevents apply. The Execution Coverage page must show simulation posture.
- **Provider write execution**: `ProviderWriteExecution` records the actual write against the RPKI provider. It links back to a change plan and optionally triggers a followup sync run. The Execution Coverage and Drift Workbench pages need this to show whether drift has actually been acted on.
- **Rollback bundles**: `ROAChangePlanRollbackBundle` captures the inverse delta for a plan that has been applied. Rollback state is a key element of Execution Coverage — it tells the operator whether a revert is possible.
- **Approval records**: `ApprovalRecord` captures disposition, recorded_by, ticket_reference, and simulation_review_json. This is the governance audit trail. The Change Review and Execution Coverage pages need it.
- **Delegated publication workflows**: `DelegatedPublicationWorkflow` represents an upstream-managed publication workflow for a downstream entity. Multi-party authority chains are invisible in the current UI.

## Main UX Problems

1. **Navigation is model-oriented, not task-oriented.** The Intent menu contains 11 CRUD lists. Operators land on object lists instead of pages that answer authority, drift, ownership, and readiness questions.

2. **There is no first-class concept of the current authoritative run-set.** Users must infer which derivation or reconciliation run is the one that matters right now. The UI shows all runs in reverse chronological order with no "this is the current authority" marker.

3. **Reconciliation review is split across three result families.** To understand one prefix, the operator pivots between `ROAIntentResult` (intent-side view), `PublishedROAResult` (published-side view), `ROAIntentMatch` (candidate match details), and run detail pages. No single row joins both sides.

4. **The execution pipeline is invisible as a pipeline.** Derivation → reconciliation → lint → plan → simulation → approval → apply → rollback is a sequence, but the UI treats each stage as an independent model family. There is no per-subject view showing which stages have completed and where things stalled.

5. **The lint quality gate is buried.** Lint findings, acknowledgements, and suppressions are accessible from change plan detail, but there is no cross-plan view of lint posture. An operator cannot quickly answer "which plans are blocked by lint?" without visiting each plan individually.

6. **ROV simulation posture is only visible from change plan detail.** An operator cannot answer "which subjects will go INVALID if this plan is applied?" without navigating into the plan, then the simulation run, then the individual results.

7. **Exposure state is not surfaced in any summary view.** `ROAIntent.exposure_state` (ADVERTISED / ELIGIBLE_NOT_ADVERTISED / BLOCKED) tells you whether a prefix with authority is actually being announced. No existing page surfaces this or filters by it.

8. **Delegated authority chains are opaque.** When a managed relationship designates a downstream entity, the authority map for that prefix involves the upstream organization, the delegated entity, the managed relationship, and optionally a delegated publication workflow. None of this is surfaced inline anywhere — the operator must follow FK links across four model detail pages.

9. **Template binding staleness is only visible in the operations dashboard.** The ops dashboard shows stale bindings, but the Intent menu pages do not. An operator reviewing authority cannot see that the underlying binding is STALE without visiting the operations dashboard or the binding detail page.

10. **ASPA intent has no structured authority explainer.** The lack of a derivation run, source profile, source rule, derived state, and structured summary means ASPA cannot support the same depth of authority explanation that ROA can. Shipping a thin ASPA version of the authority page would set incorrect parity expectations.

## Recommended Pages

### 1. Intent Authority Map

Primary question: **What is authoritative for this resource right now, and why?**

This should become the default landing page in the Intent group.

**Subject row key**: For ROA, the combination of prefix CIDR, origin ASN, and max length from the authoritative `ROAIntent`. For ASPA (phase 3), the customer ASN with the full intended provider set.

**Authority chain per row** (all available from existing `ROAIntent` fields and `summary_json`):

| Chain element | Source field(s) |
|---------------|----------------|
| Organization | `ROAIntent.organization` |
| Intent profile | `ROAIntent.intent_profile` |
| Source rule | `ROAIntent.source_rule` |
| Template binding(s) | `summary_json.binding_context_groups` keys |
| Profile context groups | `summary_json.profile_context_group_names` |
| Applied override | `ROAIntent.applied_override` |
| Delegated entity | `ROAIntent.delegated_entity` |
| Managed relationship | `ROAIntent.managed_relationship` |
| Explanation | `ROAIntent.explanation` |

**Key state dimensions per row**:

| Dimension | Source | Values |
|-----------|--------|--------|
| Derived state | `ROAIntent.derived_state` | ACTIVE, SUPPRESSED, SHADOWED |
| Exposure state | `ROAIntent.exposure_state` | ADVERTISED, ELIGIBLE_NOT_ADVERTISED, BLOCKED |
| Address family | `ROAIntent.address_family` | IPV4, IPV6 |
| AS0 flag | `ROAIntent.is_as0` | True, False |
| Run state | Derived from reconciliation join | unreconciled, reconciled_current, reconciled_with_drift, reconciliation_failed |
| Drift classification | `ROAIntentResult.result_type` | 12 values from `ROAIntentResultType` |
| Drift severity | `ROAIntentResult.severity` | INFO, WARNING, ERROR, CRITICAL |
| Comparison scope | `ROAReconciliationRun.comparison_scope` | LOCAL_ROA_RECORDS, PROVIDER_IMPORTED |
| Binding freshness | `RoutingIntentTemplateBinding.state` | CURRENT, STALE, PENDING, INVALID |
| Overlap detection | Computed from multiple rows sharing the same prefix | overlap, no_overlap |

**Filters** (all based on fields already available on the source models):

- `organization` — FK on ROAIntent
- `intent_profile` — FK on ROAIntent
- `address_family` — field on ROAIntent
- `derived_state` — field on ROAIntent (ACTIVE/SUPPRESSED/SHADOWED)
- `exposure_state` — field on ROAIntent (ADVERTISED/ELIGIBLE_NOT_ADVERTISED/BLOCKED)
- `delegated_entity` — FK on ROAIntent
- `managed_relationship` — FK on ROAIntent
- `provider_account` — FK on ROAChangePlan (downstream join)
- `authority_state` / `run_state` / `drift_state` — service-computed classifications
- `q` — text search across prefix CIDR, ASN value, profile name, rule name, explanation

**Authoritative selection rule**:

For each `RoutingIntentProfile` with `status=ACTIVE` and `enabled=True`, select the latest `IntentDerivationRun` with `status=COMPLETED`. The `ROAIntent` rows from that run are the current authority for that profile.

If two profiles produce intents for the same prefix, both rows are shown with an overlap indicator rather than attempting merge semantics. Merge requires policy decisions about profile priority that do not exist yet.

**Reconciliation join**: For each authoritative derivation run, look up the latest `ROAReconciliationRun` with `status=COMPLETED` whose `basis_derivation_run` matches. If there are multiple (e.g., one LOCAL_ROA_RECORDS and one PROVIDER_IMPORTED), the page should prefer the PROVIDER_IMPORTED run when the profile's organization has an active provider account with a completed snapshot, and fall back to LOCAL_ROA_RECORDS otherwise. Both should be accessible via a detail toggle.

**What existing data supports this**:

- Every field in the authority chain is already persisted on `ROAIntent` or in its `summary_json`.
- Reconciliation results are already linked through `basis_derivation_run`.
- The derivation service already builds the narrative explanation in `_evaluate_compiled_roa_intents()`.
- The operations dashboard template and IRR divergence dashboard template establish the summary-card + filtered-table pattern.

### 2. Intent Drift Workbench

Primary question: **Where does runtime state differ from authoritative intent, and what should the operator do about it?**

This replaces the current workflow of opening a reconciliation run, then separately inspecting intent results and published results.

**Grouping model**: Group drift rows by operator action. Map the 12 `ROAIntentResultType` values and 11 `PublishedROAResultType` values into operator-action groups:

| Group | Intent-side result types | Published-side result types | Operator action |
|-------|--------------------------|----------------------------|-----------------|
| Missing from runtime | MISSING | — | Create ROA via change plan |
| Wrong origin | ASN_MISMATCH, ASN_AND_MAX_LENGTH_OVERBROAD, ASN_AND_MAX_LENGTH_TOO_NARROW, ASN_AND_MAX_LENGTH_MISMATCH | WRONG_ORIGIN, WRONG_ORIGIN_AND_MAX_LENGTH_OVERBROAD, WRONG_ORIGIN_AND_MAX_LENGTH_TOO_NARROW, WRONG_ORIGIN_AND_MAX_LENGTH_MISMATCH | Replace ROA via change plan |
| maxLength mismatch | MAX_LENGTH_OVERBROAD, MAX_LENGTH_TOO_NARROW | BROADER_THAN_NEEDED, MAX_LENGTH_TOO_NARROW | Adjust maxLength via change plan |
| Prefix mismatch | PREFIX_MISMATCH | — | Review prefix selector |
| Stale evidence | STALE | STALE | Re-sync provider or re-run reconciliation |
| Orphaned published state | — | ORPHANED, DUPLICATE, UNSCOPED | Withdraw orphan via change plan |
| Inactive or suppressed | INACTIVE_INTENT, SUPPRESSED_BY_POLICY | — | Review override or exception |

**Each drift row should show both sides**:

- Left: authoritative intent (prefix, origin, max length, profile, explanation)
- Right: best runtime match (source ROA or imported authorization, comparison scope, provider snapshot age)
- Center: drift classification, severity, lint posture, available actions

**Evidence source awareness**: Display whether the reconciliation compared against `LOCAL_ROA_RECORDS` or `PROVIDER_IMPORTED`, and show the provider snapshot timestamp when available. Drift against provider-imported evidence is actionable immediately; drift against local records may require a sync first.

**Lint integration**: For any subject where a `ROAChangePlanItem` already exists, show the latest lint posture (clean, has-warnings, has-blocking-findings, has-acknowledged-findings). This tells the operator whether the plan is ready for approval or needs lint attention first.

**External management overlay**: When an `ExternalManagementException` covers the drifting subject, display it inline with a badge. This explains why drift may be expected and accepted.

### 3. Authority Change Review

Primary question: **What changed in authority between derivation runs, and why?**

**Diff model**: Compare the `ROAIntent` set from the current authoritative derivation run against the prior completed derivation run for the same profile. Group changes by reason:

| Change reason | Detection method |
|---------------|-----------------|
| Selector scope changed | `IntentDerivationRun.input_fingerprint` differs between runs |
| Template policy changed | `RoutingIntentTemplateBinding.state` is STALE or template `template_fingerprint` changed |
| Binding scope changed | Binding's `prefix_selector_query` or `asn_selector_query` changed |
| Override added or removed | `ROAIntent.applied_override` changed between runs |
| Exception influence changed | `RoutingIntentException` enabled/disabled or expired between runs |
| Delegated scope resolved differently | `summary_json.delegated_scope` differs between runs |
| Profile rule changed origin or max length | `ROAIntent.source_rule` or intent values differ |
| Subject appeared or disappeared | Intent key present in one run but not the other |

**Before/after chain**: Show the full authority chain for both the old and new row so the operator can see not just "the origin changed" but "the origin changed because rule X was replaced by rule Y after template binding Z was regenerated."

**Entry points**: Reachable from profile detail (show changes for this profile), bulk run detail (show changes introduced by this bulk run), and the Authority Map (show how this row's authority was established).

**Fingerprint leverage**: `IntentDerivationRun.input_fingerprint`, `BulkIntentRun.baseline_fingerprint` / `resulting_fingerprint`, and `RoutingIntentTemplate.template_fingerprint` already exist. The service layer can use fingerprint comparison to detect whether inputs actually changed between runs.

### 4. Intent Execution Coverage

Primary question: **Has each subject completed the full workflow pipeline, or did a stage stall?**

**Pipeline stages per subject**:

| Stage | Source model | Completeness indicator |
|-------|-------------|----------------------|
| Authority compiled | `IntentDerivationRun` | status=COMPLETED, has ROAIntent rows |
| Reconciled | `ROAReconciliationRun` | status=COMPLETED, has ROAIntentResult |
| Drift classified | `ROAIntentResult` | result_type populated |
| Lint passed | `ROALintRun` | status=COMPLETED, finding_count / critical_count / error_count |
| Plan generated | `ROAChangePlan` | Exists for source reconciliation |
| Lint acknowledged | `ROALintAcknowledgement` | All blocking findings acknowledged |
| ROV simulation passed | `ROAValidationSimulationRun` | overall_approval_posture != BLOCKING |
| Governance approved | `ROAChangePlan.approved_at` | Non-null, or secondary_approved_at for dual-approval plans |
| Applied | `ProviderWriteExecution` | status=COMPLETED |
| Verified | `ProviderWriteExecution.followup_sync_run` | Followup sync and snapshot exist |
| Rollback available | `ROAChangePlanRollbackBundle` | status=AVAILABLE or APPROVED |

**Actionable filters**:

- "Never reconciled" — authority exists but no reconciliation run matches
- "Reconciled but no plan" — drift exists but no change plan was created
- "Plan blocked by lint" — change plan exists with unresolved blocking lint findings
- "Plan blocked by simulation" — simulation run has overall_approval_posture=BLOCKING
- "Approved but not applied" — plan approved but no write execution
- "Applied but not verified" — write execution completed but no followup sync
- "Failed" — write execution or reconciliation with status=FAILED

**Governance metadata**: Show `ticket_reference`, `change_reference`, `maintenance_window_start/end`, `approved_by`, `secondary_approved_by` inline for plans requiring governance audit.

### 5. Overrides And Exceptions Workbench

Primary question: **What is distorting baseline policy, and is that distortion still justified?**

**Combined view across four exception types**:

| Exception type | Model | Temporal fields | Review triggers |
|----------------|-------|----------------|-----------------|
| ROA intent overrides | `ROAIntentOverride` | `starts_at`, `ends_at`, `enabled` | Expired, disabled, masking drift |
| Routing intent exceptions | `RoutingIntentException` | `starts_at`, `ends_at`, `approved_by`, `enabled` | Expired, unapproved, masking drift |
| External management exceptions | `ExternalManagementException` | `starts_at`, `review_at`, `ends_at`, `approved_by`, `enabled` | Review-due, expired, masking drift |
| Lint suppressions | `ROALintSuppression` | `created_at`, `expires_at`, `lifted_at`, `last_matched_at` | Expired, never matched, lifted |

**Grouping by review reason**:

- **Expired or expiring**: `ends_at` or `expires_at` in the past, or within a configurable horizon
- **Review-due**: `ExternalManagementException.review_at` in the past
- **Unapproved**: `RoutingIntentException.approved_by` blank, or `ExternalManagementException.approved_by` blank
- **Currently masking drift**: Cross-reference with latest reconciliation results to show when an exception covers a subject that would otherwise show as drifted
- **Never matched**: `ROALintSuppression.match_count = 0` — suppression may be stale
- **No longer matched by current authority**: Override or exception targets a prefix or ASN that no longer appears in the current authoritative intent set

**Subject linkage**: For each exception, show the affected subjects (prefixes or ASNs) rather than just the exception object. This requires joining against the latest authoritative intent set.

## Data And Service Changes Required

### Changes that use only existing data (no schema changes)

1. **Authority selection service**: Query `IntentDerivationRun` for the latest completed run per active+enabled profile. Load `ROAIntent` rows from those runs. This requires only existing indexes (`nb_rpki_ri_org_prof_idx`).

2. **Reconciliation join service**: For each authoritative derivation run, find the latest completed `ROAReconciliationRun` via the existing `basis_derivation_run` FK. Load `ROAIntentResult` rows indexed by `roa_intent_id`.

3. **Subject-centric row projection**: Parse `ROAIntent.summary_json` for context groups, warnings, and delegated scope. Parse `ROAIntentResult.details_json` for mismatch axes. Derive operator-friendly summary strings. No new model fields needed.

4. **Overlap detection**: Identify ROAIntent rows from different profiles that cover the same prefix. This is a set intersection on `prefix_cidr_text` across the authoritative intent set.

5. **Drift grouping**: Map `ROAIntentResultType` enum values and `PublishedROAResultType` enum values into operator-action groups. This is a pure service-layer mapping.

6. **Exception cross-reference**: Load active `ROAIntentOverride`, `RoutingIntentException`, `ExternalManagementException`, and `ROALintSuppression` rows. Index by prefix or subject key. Flag exceptions that match subjects in the current authority set.

### Model gaps requiring schema changes in later phases

1. **ASPA provenance parity** (phase 3): Add to `ASPAIntent`:
   - `derivation_run` FK to a new `ASPADerivationRun` model
   - `intent_profile` FK (or shared profile model)
   - `source_rule` FK
   - `derived_state` field
   - `summary_json` structured field
   This is the prerequisite for extending the Authority Map to ASPA.

2. **Explicit authority snapshot** (phase 2): Add a model or a soft marker that identifies which derivation run is the "current authority" for a profile, rather than inferring it from latest-completed. This enables pinning authority to a specific run during incident response.

3. **Subject-level stable key**: `ROAIntent.intent_key` already exists as a SHA-256 hash. Promote it as the cross-model join key for subject-level aggregation. Ensure reconciliation results, change plan items, and lint findings can be joined back to the same key without expensive prefix+origin+maxlength triple lookups.

### Useful service-layer projections (no schema changes)

| Projection | Derivation |
|------------|-----------|
| `authority_reason_summary` | Condensed `ROAIntent.explanation` + override/exception influence |
| `runtime_evidence_summary` | Best match description from `ROAIntentResult.details_json` + comparison scope + snapshot age |
| `lint_posture` | Roll up from latest `ROALintRun` per reconciliation: clean / warnings / blocking / acknowledged |
| `simulation_posture` | From latest `ROAValidationSimulationRun` per change plan: informational / acknowledgement_required / blocking |
| `execution_stage` | Derived from which pipeline models exist and their statuses |
| `binding_freshness` | From `RoutingIntentTemplateBinding.state` for bindings referenced in `summary_json` |
| `follow_up_action` | Determined by drift classification + lint posture + plan status + exception coverage |

## Delivery Plan

### Phase 1: ROA Authority Map and Drift Workbench (read-only)

**Scope**:
- Implement `build_roa_authority_map()` service with authority selection, reconciliation join, overlap detection, and row projection
- Implement `build_roa_drift_workbench()` service that groups authority map rows by drift action category
- Add `IntentAuthorityMapView` and `IntentDriftWorkbenchView` as standalone dashboard views
- Add filter forms, table classes, and templates following IRR divergence dashboard pattern
- Add navigation entries at positions 1 and 2 in the Intent group
- Add URL routes, view tests, navigation tests, and service tests

**Prerequisites**: None. All data already exists.

**Expected impact**: Immediate improvement in discoverability and triage. Operators can answer "what is authoritative?" and "where is drift?" from single pages instead of assembling the answer across multiple model lists.

**Risk**: Performance on large intent sets (10,000+ rows per derivation run). Mitigate with server-side pagination and bulk indexed lookups.

### Phase 2: Authority Change Review, Execution Coverage, Overrides Workbench

**Scope**:
- Implement authority diff service comparing consecutive derivation runs
- Implement pipeline coverage service aggregating derivation → reconciliation → lint → plan → simulation → approval → apply → verify stages per subject
- Implement combined exception/override review service with cross-reference against current authority
- Add views, forms, tables, templates, navigation, routes, and tests for all three pages

**Prerequisites**: Phase 1 services (reuse authority selection and reconciliation join). Explicit authority snapshot model (or decision to continue with latest-completed inference).

**Expected impact**: Operators can explain changes ("why did this prefix change authority?"), find stalled workflows ("which plans are blocked by lint?"), and review exceptions ("which overrides are expired and still masking drift?").

### Phase 3: ASPA provenance parity

**Scope**:
- Add `ASPADerivationRun` model (or extend `IntentDerivationRun`)
- Add provenance fields to `ASPAIntent` (derivation_run, intent_profile, source_rule, derived_state, summary_json)
- Build ASPA derivation service parallel to ROA
- Extend Authority Map and Drift Workbench to support ASPA subjects

**Prerequisites**: Phase 1 and 2 complete. Schema migration plan agreed.

**Expected impact**: Consistent mental model across ROA and ASPA instead of a ROA-only premium path.

### Phase 4: Workflow actions from operator pages

**Scope**:
- Add action buttons to Authority Map: "Run Reconciliation", "View Profile"
- Add action buttons to Drift Workbench: "Create Change Plan", "View Reconciliation Run"
